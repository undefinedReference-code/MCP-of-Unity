# Unity MCP Min：运行指南（中文）

本文说明如何在本仓库中**跑通原型**、MCP 层**暴露了哪些工具**及用法，以及 **Skill（技能）** 如何引导模型去调用这些工具。

---

## 一、项目是什么

这是一个**最小可行原型（MVP）**：

- **MCP 侧**：用 **Python** 启动 MCP Server，对外只暴露 **3 个工具**。
- **Unity 侧**：通过 **`com.mcpofunity.reflect-bridge`** 包提供开箱即用的 **HTTP 桥接 + 配置窗口**（内部调用 `ReflectCallTool`）。

当前 MVP 聚焦三类任务：

1. **Transform**：例如设置 `position`、`eulerAngles` 等（仍为同一工具 + 不同 `member`/参数）。
2. **创建物体**：对 **`GameObject` 的静态方法** 做 `invoke`（例如 `CreatePrimitive`），**不需要** `objectSelector`。
3. **挂载脚本**：对实例 **`invoke`** `AddComponent`，需 `objectSelector`；参数为脚本类型名字符串。

---

## 二、运行前准备

| 依赖 | 说明 |
|------|------|
| Python | 建议 3.11+ |
| Unity | 2021.3 及以上（包依赖 Newtonsoft JSON） |
| 本仓库 Unity 包 | `prototype/unity-mcp-min/Packages/com.mcpofunity.reflect-bridge` |

## 三、Unity 侧：安装自带 HTTP 桥接（推荐）

### 3.1 把包加入 Unity 工程

在工程根目录的 `Packages/manifest.json` 的 `dependencies` 中增加（**路径按本机修改**）：

```json
"com.mcpofunity.reflect-bridge": "file:你的路径/MCP-of-Unity/prototype/unity-mcp-min/Packages/com.mcpofunity.reflect-bridge"
```

详细说明见包内：`Packages/com.mcpofunity.reflect-bridge/Documentation~/INSTALL.zh.md`。

### 3.2 启动桥接服务

1. Unity 菜单：**Window → MCP Reflect Bridge**  
2. 设置 **Listen port**（默认 **7890**，与 Python 侧 `UNITY_BRIDGE_ENDPOINT` 一致）  
3. 可选：**Auto-start when Unity opens**  
4. 点击 **Start server**

自检：浏览器访问 `http://127.0.0.1:7890/health`，应返回 JSON。

### 3.3 HTTP 约定（与 Python 一致）

Python 里的 `unity_reflect_call` 会向 Unity 发：

| 项目 | 说明 |
|------|------|
| 根地址 | `UNITY_BRIDGE_ENDPOINT`，默认 `http://127.0.0.1:7890` |
| 路径 | `POST /unity_reflect_call` |
| Body | JSON，字段见下文「`unity_reflect_call`」 |
| 自检 | `GET /health` |

桥接在 **Editor 主线程** 调用 `ReflectCallTool.HandleCommand`，避免线程安全问题。

> **自定义集成**：若你已有其它 MCP Unity 插件，也可自行注册路由并调用 `UnityMcpMin.Editor.Tools.ReflectCallTool.HandleCommand`，无需使用本包 HTTP 实现。

## 四、Python MCP Server：安装与启动

在仓库根目录下：

```bash
cd prototype/unity-mcp-min/Server
pip install -r requirements.txt
python main.py
```

默认会以 **stdio** 方式运行 MCP（符合多数 MCP Host 的对接方式）。若你的客户端要求固定工作目录，请把 `cwd` 指到 `Server` 目录。

### 环境变量

| 变量 | 含义 | 默认值 |
|------|------|--------|
| `UNITY_BRIDGE_ENDPOINT` | Unity HTTP 桥接根 URL（不含路径） | `http://127.0.0.1:7890` |

## 五、在 Cursor / Claude Desktop 里挂载 MCP（示例）

在 MCP 配置里增加一个 server，**command** 指向你的 Python 解释器，**args** 指向 `main.py`，**cwd** 为 `prototype/unity-mcp-min/Server`。

示例（路径请按本机修改）：

```json
{
  "mcpServers": {
    "unity-mcp-min": {
      "command": "python",
      "args": ["D:/C++Code/MCP-of-Unity/prototype/unity-mcp-min/Server/main.py"],
      "cwd": "D:/C++Code/MCP-of-Unity/prototype/unity-mcp-min/Server",
      "env": {
        "UNITY_BRIDGE_ENDPOINT": "http://127.0.0.1:7890"
      }
    }
  }
}
```

保存后重启客户端或重新加载 MCP，确认工具列表里出现下面三个名称。

## 六、MCP 层提供的工具（共 3 个）

下面说的「工具」都是 **MCP `tools/list` 里暴露给模型的接口**，由模型在对话中按需调用。

### 1. `unity_reflect_call`（核心：唯一真正打到 Unity 的执行入口）

**作用**：把一次「反射语义」的请求转发到 Unity，由 `ReflectCallTool` 做 **get / set / invoke**。

**主要参数**：

| 参数 | 类型 | 必填 | 说明 |
|------|------|------|------|
| `mode` | 字符串 | 是 | `get` \| `set` \| `invoke` |
| `targetType` | 字符串 | 是 | CLR 类型名，如 `UnityEngine.Transform`、`UnityEngine.GameObject` |
| `member` | 字符串 | 是 | 属性名、字段名或方法名，如 `position`、`AddComponent` |
| `args` | 数组 | 否 | `set` 一般 1 个元素；`invoke` 按方法参数个数；`get` 可省略 |
| `objectSelector` | 对象 | **仅实例成员必填** | 静态方法（如 `CreatePrimitive`）**不要传**此字段。实例成员使用 `gameObjectName` 或 `instanceId` |
| `dryRun` | 布尔 | 否 | `true` 时只做签名/绑定校验，**不执行**写操作或 invoke |

**典型用法**：

- **改位置**：`mode=set`，`targetType=UnityEngine.Transform`，`member=position`，`args=[{"x":0,"y":2,"z":0}]`，`objectSelector={"gameObjectName":"DemoCube"}`。
- **新建基础几何体（无现有物体）**：`mode=invoke`，`targetType=UnityEngine.GameObject`，`member=CreatePrimitive`，`args=["Cube"]`，**不传** `objectSelector`（静态方法）。Primitive 名称与 Unity `PrimitiveType` 一致（如 `Sphere`、`Plane`）。
- **挂脚本**：`mode=invoke`，`targetType=UnityEngine.GameObject`，`member=AddComponent`，`args=["你的命名空间.你的Behaviour类名"]`，`objectSelector` 指向目标物体。

**返回**：JSON 对象；成功时通常含 `ok: true`；失败时含 `ok: false` 与可读 `message`（便于模型自修正）。

---

### 2. `unity_schema_hint`（参数模板提示，不打 Unity）

**作用**：根据**领域**和**自然语言意图**，从本地知识库（`Server/knowledge/*.json`）里挑出最相关的 **最多 3 条** `requestTemplate`，指导模型如何填 `unity_reflect_call`。

**参数**：

| 参数 | 类型 | 必填 | 说明 |
|------|------|------|------|
| `domain` | 字符串 | 是 | `transform`、`scripting`、`scene` |
| `intent` | 字符串 | 是 | 用户任务描述，长度 ≥ 3 |

**返回**：含 `suggestions[]`，每项里有可直接拷贝再改的 `requestTemplate`（里面已是合法的 `unity_reflect_call` 参数形状）。

---

### 3. `unity_validate_result`（轻量验收，不打 Unity）

**作用**：根据上一次 `unity_reflect_call`（或其它步骤）的返回里的 `ok` 字段，给出 **accept** 或 **retry_or_fail**，方便模型决定是否重试。

**参数**：

| 参数 | 类型 | 必填 |
|------|------|------|
| `task` | 字符串 | 是 |
| `result` | 对象 | 是（通常传入上一步完整返回） |

---

### 工具之间的关系（给模型用的调用顺序）

推荐固定顺序（也与 Skill 一致）：

1. `unity_schema_hint` → 拿到模板  
2. `unity_reflect_call`（`dryRun=true`）→ 确认参数与签名  
3. `unity_reflect_call`（`dryRun=false`）→ 真正执行  
4. `unity_validate_result` → 判断是否接受结果

## 七、Skill（技能）提供了什么、和 MCP 工具的关系

本仓库里的 Skill 是 **给 Agent / 模型阅读的工作流说明**，路径为：

`prototype/unity-mcp-min/skills/unity-reflect/SKILL.md`

另有三份按需加载的参考：

- `prototype/unity-mcp-min/skills/unity-reflect/references/transform.md`
- `prototype/unity-mcp-min/skills/unity-reflect/references/scripting.md`
- `prototype/unity-mcp-min/skills/unity-reflect/references/scene.md`

### Skill 不是 MCP 协议里的「第二种工具」

- **MCP 工具**：Host 通过 MCP 暴露给模型，模型可以**直接调用**（`unity_reflect_call` 等）。
- **Skill**：通常是 **文档型引导**（何时用、按什么顺序调 MCP、出错如何改）。是否自动加载取决于你用的客户端是否支持 Agent Skills / 规则引用该目录。

### 按「渐进披露」理解 Skill 的三层

| 层级 | 内容 | 模型应怎么用 |
|------|------|----------------|
| **L1（元信息）** | `SKILL.md` 前半：适用场景（改 Transform、创建物体、挂脚本） | 判断当前用户任务是否该走「反射 + 模板」这条链路 |
| **L2（流程）** | `SKILL.md` 中的固定步骤 | **严格按顺序**调用 MCP：`unity_schema_hint` → `unity_reflect_call`(dryRun) → `unity_reflect_call`(执行) → `unity_validate_result`；创建物体后可用返回的 `instanceId` 做后续步骤 |
| **L3（参考）** | `references/*.md` | 仅在 L2 失败或需要细节时阅读：`scene.md`（无 selector 的静态创建）、`transform.md`、`scripting.md` |

**一句话**：Skill 负责教模型「**什么时候**、**按什么顺序**、**用什么参数形状**」去按那 **3 个 MCP 工具** 干活；真正执行 Unity 的只有 **`unity_reflect_call`**。

## 九、常见问题

**Q：创建物体时没有现有物体名，也能做吗？**  
A：可以。对 **`GameObject` 的静态方法**（如 **`CreatePrimitive`**）使用 **`invoke`**，并且 **不要传 `objectSelector`**。创建完成后从返回 JSON 里取 **`instanceId`**，后续再对该物体做 Transform / AddComponent。

**Q：为什么调用 `unity_reflect_call` 返回 bridge 失败？**  
A：检查 Unity 是否在跑、是否已在 **Window → MCP Reflect Bridge** 中 **Start server**、端口是否与 `UNITY_BRIDGE_ENDPOINT` 一致、路径是否为 `POST /unity_reflect_call`。

**Q：Skill 放进 Cursor 后没有自动生效？**  
A：需在 Cursor 的 Skills / 规则里**指向该 Skill 目录或文件**；不同版本配置入口名称可能不同，以当前客户端文档为准。

**Q：能否用这一套点编辑器里每一个按钮？**  
A：当前 MVP 针对 **引擎对象 API 反射**，不覆盖全部 Editor UI。要点菜单等需另行封装（例如 `ExecuteMenuItem`）或 UI 自动化，属于扩展话题。

---

如需扩展更多领域，可在 `Server/knowledge/` 增加新的 `*.json` 并在 Python 的 `SchemaHintRequest` 中放开 `domain` 枚举（当前仅 `transform` / `scripting`）。
