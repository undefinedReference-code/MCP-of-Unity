# Unity MCP Min：运行指南（中文）

本文说明如何在本仓库中**跑通原型**、MCP 层**暴露了哪些工具**及用法，以及 **Skill（技能）** 如何引导模型去调用这些工具。

---

## 一、项目是什么

这是一个**最小可行原型（MVP）**：

- **MCP 侧**：用 **Python** 启动 MCP Server，对外只暴露 **3 个工具**。
- **Unity 侧**：用 **C#** 在编辑器里执行反射调用（本仓库提供 `ReflectCallTool`，不负责完整的 Unity HTTP 插件；你需要在自己的 Editor 桥接里把 HTTP 请求转发到它）。

当前 MVP 聚焦两类任务：

1. **Transform**：例如设置 `position`、`localPosition`、`eulerAngles` 等（**不设**「单独的位置工具」「单独的旋转工具」）。
2. **挂载脚本**：对 `GameObject` **invoke** `AddComponent`，参数里传**脚本类型名**（解析为 `System.Type`）。

## 三、Unity 侧：桥接接口（必须实现）

Python 里的 `unity_reflect_call` 会向 Unity 发 **HTTP POST**：

- **默认地址**：环境变量 `UNITY_BRIDGE_ENDPOINT`，缺省为 `http://127.0.0.1:7890`
- **路径**：`/unity_reflect_call`
- **Body**：JSON，字段与下面「`unity_reflect_call`」一节一致。

Unity 端需要在收到请求后：

1. 把 JSON 转成 `Newtonsoft.Json.Linq.JObject`（或与现有桥接一致的结构）。
2. 调用本仓库中的：

   `UnityMcpMin.Editor.Tools.ReflectCallTool.HandleCommand(JObject parameters)`

3. 将返回结果序列化为 JSON 写回 HTTP 响应。

> **说明**：本仓库**没有**附带完整的 Unity Package（例如自动起监听端口、路由注册）。若你已有其他 Unity MCP 的 Editor 插件，只需增加上述路由并接入 `HandleCommand` 即可。

## 四、Python MCP Server：安装与启动

在仓库根目录下：

```bash
cd prototype/unity-mcp-min/Server
pip install -r requirements.txt
python main.py
```

默认会以 **stdio** 方式运行 MCP（符合多数 MCP Host 的对接方式）。若你的客户端要求固定工作目录，请把 `cwd` 指到 `Server` 目录。

### 环境变量

| 变量 | 含义 | 默认值 |
|------|------|--------|
| `UNITY_BRIDGE_ENDPOINT` | Unity HTTP 桥接根 URL（不含路径） | `http://127.0.0.1:7890` |

在 MCP 配置里增加一个 server，**command** 指向你的 Python 解释器，**args** 指向 `main.py`，**cwd** 为 `prototype/unity-mcp-min/Server`。

## 六、MCP 层提供的工具（共 3 个）

下面说的「工具」都是 **MCP `tools/list` 里暴露给模型的接口**，由模型在对话中按需调用。

### 1. `unity_reflect_call`（核心：唯一真正打到 Unity 的执行入口）

**作用**：把一次「反射语义」的请求转发到 Unity，由 `ReflectCallTool` 做 **get / set / invoke**。

**主要参数**：

| 参数 | 类型 | 必填 | 说明 |
|------|------|------|------|
| `mode` | 字符串 | 是 | `get` \| `set` \| `invoke` |
| `targetType` | 字符串 | 是 | CLR 类型名，如 `UnityEngine.Transform`、`UnityEngine.GameObject` |
| `member` | 字符串 | 是 | 属性名、字段名或方法名，如 `position`、`AddComponent` |
| `args` | 数组 | 否 | `set` 一般 1 个元素；`invoke` 按方法参数个数；`get` 可省略 |
| `objectSelector` | 对象 | 实例成员时必填 | `gameObjectName` 或 `instanceId`（二选一或按你桥接约定） |
| `dryRun` | 布尔 | 否 | `true` 时只做签名/绑定校验，**不执行**写操作或 invoke |

**典型用法**：

- **改位置**：`mode=set`，`targetType=UnityEngine.Transform`，`member=position`，`args=[{"x":0,"y":2,"z":0}]`，`objectSelector={"gameObjectName":"DemoCube"}`。
- **挂脚本**：`mode=invoke`，`targetType=UnityEngine.GameObject`，`member=AddComponent`，`args=["你的命名空间.你的Behaviour类名"]`，`objectSelector` 同上。

**返回**：JSON 对象；成功时通常含 `ok: true`；失败时含 `ok: false` 与可读 `message`（便于模型自修正）。

---

### 2. `unity_schema_hint`（参数模板提示，不打 Unity）

**作用**：根据**领域**和**自然语言意图**，从本地知识库（`Server/knowledge/*.json`）里挑出最相关的 **最多 3 条** `requestTemplate`，指导模型如何填 `unity_reflect_call`。

**参数**：

| 参数 | 类型 | 必填 | 说明 |
|------|------|------|------|
| `domain` | 字符串 | 是 | 当前仅支持：`transform`、`scripting` |
| `intent` | 字符串 | 是 | 用户任务描述，长度 ≥ 3 |

**返回**：含 `suggestions[]`，每项里有可直接拷贝再改的 `requestTemplate`（里面已是合法的 `unity_reflect_call` 参数形状）。

---

### 3. `unity_validate_result`（轻量验收，不打 Unity）

**作用**：根据上一次 `unity_reflect_call`（或其它步骤）的返回里的 `ok` 字段，给出 **accept** 或 **retry_or_fail**，方便模型决定是否重试。

**参数**：

| 参数 | 类型 | 必填 |
|------|------|------|
| `task` | 字符串 | 是 |
| `result` | 对象 | 是（通常传入上一步完整返回） |

---

### 工具之间的关系（给模型用的调用顺序）

推荐固定顺序（也与 Skill 一致）：

1. `unity_schema_hint` → 拿到模板  
2. `unity_reflect_call`（`dryRun=true`）→ 确认参数与签名  
3. `unity_reflect_call`（`dryRun=false`）→ 真正执行  
4. `unity_validate_result` → 判断是否接受结果

## 七、Skill（技能）提供了什么、和 MCP 工具的关系

本仓库里的 Skill 是 **给 Agent / 模型阅读的工作流说明**，路径为：

`prototype/unity-mcp-min/skills/unity-reflect/SKILL.md`

另有两份按需加载的参考：

- `prototype/unity-mcp-min/skills/unity-reflect/references/transform.md`
- `prototype/unity-mcp-min/skills/unity-reflect/references/scripting.md`

### Skill 不是 MCP 协议里的「第二种工具」

- **MCP 工具**：Host 通过 MCP 暴露给模型，模型可以**直接调用**（`unity_reflect_call` 等）。
- **Skill**：通常是 **文档型引导**（何时用、按什么顺序调 MCP、出错如何改）。是否自动加载取决于你用的客户端是否支持 Agent Skills / 规则引用该目录。

### 按「渐进披露」理解 Skill 的三层

| 层级 | 内容 | 模型应怎么用 |
|------|------|----------------|
| **L1（元信息）** | `SKILL.md` 前半：适用场景（改 Transform、挂脚本） | 判断当前用户任务是否该走「反射 + 模板」这条链路 |
| **L2（流程）** | `SKILL.md` 中的固定步骤 | **严格按顺序**调用 MCP：`unity_schema_hint` → `unity_reflect_call`(dryRun) → `unity_reflect_call`(执行) → `unity_validate_result`，失败时按说明做一次重试 |
| **L3（参考）** | `references/*.md` | 仅在 L2 失败或需要细节时阅读：例如嵌套类型写法、`Vector3` JSON 形状、`AddComponent` 的类型名字符串 |

**一句话**：Skill 负责教模型「**什么时候**、**按什么顺序**、**用什么参数形状**」去按那 **3 个 MCP 工具** 干活；真正执行 Unity 的只有 **`unity_reflect_call`**。

## 九、常见问题

**Q：为什么调用 `unity_reflect_call` 返回 bridge 失败？**  
A：检查 Unity 是否在跑、桥接是否监听 `UNITY_BRIDGE_ENDPOINT`、路径是否为 `POST /unity_reflect_call`，以及是否调用了 `ReflectCallTool.HandleCommand`。

**Q：Skill 放进 Cursor 后没有自动生效？**  
A：需在 Cursor 的 Skills / 规则里**指向该 Skill 目录或文件**；不同版本配置入口名称可能不同，以当前客户端文档为准。

**Q：能否用这一套点编辑器里每一个按钮？**  
A：当前 MVP 针对 **引擎对象 API 反射**，不覆盖全部 Editor UI。要点菜单等需另行封装（例如 `ExecuteMenuItem`）或 UI 自动化，属于扩展话题。

---

如需扩展更多领域，可在 `Server/knowledge/` 增加新的 `*.json` 并在 Python 的 `SchemaHintRequest` 中放开 `domain` 枚举（当前仅 `transform` / `scripting`）。
